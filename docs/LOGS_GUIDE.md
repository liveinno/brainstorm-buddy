# Как читать логи BrainstormBuddy — справка

## Где лежат логи

| Что | Где |
|---|---|
| Основной лог | `%APPDATA%\BrainstormBuddy\logs\app.log` |
| Ротация (по 1 МБ, 5 файлов) | `app.1.log` … `app.4.log` рядом (1 — самый свежий из старых) |
| **«Хлебная крошка»** — где лежат файлы ИМЕННО текущего запуска | `C:\ProgramData\BrainstormBuddy\last-session.txt` — строка на каждый старт: время, PID, версия, exe, реальные пути AppData/лога. **Начинать поиск всегда отсюда**: если процесс работает под файловой виртуализацией (MSIX/sandbox), его «%APPDATA%» может оказаться контейнерной папкой — крошка покажет правду. |
| Резервный лог | `%TEMP%\BrainstormBuddy\logs\app.log` — если основной путь недоступен (3 отказа записи подряд), логгер сам переезжает сюда и помечает аварию |
| Диагностический архив | Настройки → Диагностика → «Собрать логи» → zip в `Документы\BrainstormBuddy` (логи + обезличенные настройки + сведения о системе) |

Тумблеры: Настройки → Диагностика — «Вести лог-файл» (вкл/выкл), «Подробный лог» (verbose —
включает Debug-уровень: покадровые VAD-события, фрагменты распознанной речи).

## Формат строки

```
[19:52:46.156] [Info ] [Worker    ] STT [loopback] done in 164ms: 'светить по'
 └─ время      └─ уровень └─ категория └─ сообщение
```

Уровни: `Debug` (только при verbose), `Info`, `Warn`, `Error`.

## Категории — кто пишет

| Категория | Подсистема |
|---|---|
| `App` | старт/останов, пути, окружение |
| `Config` | загрузка/миграция/сохранение конфига |
| `Audio` | захват (WASAPI/микрофон), VAD, нарезка, адаптивный эндпойнтинг |
| `Loop` | главный цикл: забор готовых чанков → очередь STT |
| `Worker` | STT-воркер: распознавание чанков |
| `Answer` | воркер ответов LLM |
| `Ai` | HTTP к LLM/STT: запросы, коды, health-статусы |
| `UI` | окна, кнопки, настройки |
| `Log` | сам логгер (аварии записи, переезд на резервный путь) |

## Ключевые строки по этапам конвейера

### 1. Старт — «паспорт сессии»
```
[App   ] === BrainstormBuddy starting ===
[App   ] AppData dir: C:\Users\...\Roaming\BrainstormBuddy      ← куда реально пишет ЭТОТ запуск
[App   ] Config path: ...\config.json
[Audio ] Audio pipeline created (VAD=True, DualBuffer=True, endpoint=adaptive,
         autoCal=True, rms=0,0100, minSpeech=400ms, vadMode=0)   ← ВЕСЬ аудио-конфиг сессии одной строкой
[Audio ] WASAPI loopback started / Microphone started: device #0
```
`DualBuffer=True` = два независимых канала: `[loopback]` — динамик (собеседник), `[mic]` — микрофон.

### 2. Нарезка звука (Audio)
```
[Audio ] VAD-Level: state=SPEECH rms=0.0123 thr=0.0034 (auto,floor=0.0010)
         active=4.2s preRoll=0.40s silFrames=12/60 frames=1500 speech%=43%
```
Пишется каждые 5с. Читается так: `state` — буфер сейчас «в речи»/«в тишине»; `rms` — текущий
уровень; `thr` — действующий порог (в скобках — авто-калибровка и её шумовой пол); `active` —
сколько звука накоплено в открытой фразе; `silFrames=12/60` — счётчик тишины: 12 тихих кадров
из 60 нужных для закрытия фразы; `speech%` — доля «речевых» кадров с начала сессии.

**Диагноз по VAD-Level:** уровень скачет (`rms` живой), а `silFrames` вечно `0/60` и фраза не
закрывается → звук не опускается ниже порога (сплошной звук или порог слишком низкий).
`rms` ниже `thr` во время речи → порог выше голоса (речь не ловится).

```
[Audio ] Adaptive endpoint → 1,57s (frames=52, p85=1890ms, n=15)
```
«Авто по паузам» пересчитал порог тишины под темп говорящего (p85 — 85-й перцентиль пауз, n — выборка).

```
[Audio ] VAD: silence emission (...)   ← фраза закрыта ПАУЗОЙ — норма
[Audio ] VAD: latch emission (..., aboveThrRun=334, ...)  ← 10с сплошного звука без пауз (пожарный выход)
[Audio ] VAD: forced emission (...)    ← потолок 60с (почти недостижим при живом латче)
[Audio ] VAD: suppressed too-short utterance (speech_ms=180, min=400)  ← фраза короче MinSpeech — ВЫБРОШЕНА
```

### 3. Чанк ушёл в распознавание (Loop → Worker)
```
[Loop  ] Chunk #20 [loopback] enqueued: 749804 bytes    ← кусок звука встал в очередь STT
[Worker] STT [loopback] done in 1592ms: 'ну наверное основное я уже...'
```
`bytes / 32000 ≈ секунды звука` (16кГц, 16бит). Нет строк `Chunk #N [mic]` при живом микрофоне —
микрофонный канал не нарезает (смотри VAD-Level и пороги), это ДО распознавания.

### 4. LLM (Ai / Answer)
```
[Ai    ] CheckLlm chat[tencent/hy3:free] ← 200      ← проверка связи прошла
[Ai    ] Health[Llm] → OK: LLM-сервер отвечает       ← плашка здоровья в оверлее
[Answer] ...                                          ← ответы/ошибки реальных запросов
```

## Рецепты: симптом → что смотреть

| Симптом | Порядок диагностики |
|---|---|
| **Столбики уровня есть, текста нет** | 1) Есть ли `Chunk #N enqueued`? НЕТ → нарезка: смотри `VAD-Level` (rms vs thr; silFrames застрял на 0 → сплошной звук/низкий порог; ждём `latch emission`) и `suppressed too-short`. 2) Чанки есть, а `STT done` пустой/нет → движок распознавания. 3) `STT done` с текстом, а на экране пусто → пост-фильтр текста (см. ниже) или LLM. |
| **Короткие фразы («да», «ага») пропадают** | Ищи в verbose `only N meaningful words` / `too short` — текст выброшен пост-фильтром `TextPostProcessor` (стоп-лист паразитов + минимум 3 значимых слова). |
| **Реплика с микрофона не появилась** | grep `\[mic\]` — были ли вообще mic-чанки; если нет — VAD mic-канала (пороги, MinSpeech, фраза не закрылась из-за фонового звука). |
| **«Проверить подключение» ОК, а ответы с ошибкой** | Сравни `CheckLlm ← 200` и соседние ошибки реальных запросов — у проверки и боевого запроса могут отличаться параметры (лимит токенов, промпт). |
| **Звук с динамика пропал после смены устройства (BT-наушники)** | Смотри, было ли после подключения повторное `WASAPI loopback started` / `Loopback format` — захват привязывается к устройству на старте. |
| **Логов «нет»** | `C:\ProgramData\BrainstormBuddy\last-session.txt` → строка последнего запуска → поле `log=` — реальный путь. Затем `%TEMP%\BrainstormBuddy\logs` (резерв) и категория `Log` (аварии записи). |
| **Приложение исчезло/упало** | Конец лога: `watchdog`/последние строки; `Журнал событий Windows → Application` (.NET Runtime / Application Error). |

## Быстрые greps

```bash
grep "Chunk #"        app.log   # все нарезанные куски (канал, размер, время)
grep "STT \["         app.log   # все распознавания с длительностью и текстом
grep "VAD-Level"      app.log   # уровень/порог/счётчик тишины каждые 5с
grep -E "latch|forced|suppressed" app.log  # аварийные/подавленные эмиты
grep -E "Health|CheckLlm|Error"   app.log  # здоровье LLM и ошибки
grep "starting ==="   app.log   # границы сессий в файле
```
