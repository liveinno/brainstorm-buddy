# Бенчмарк STT: Docker vs нативный ONNX (CPU/GPU)

Реальный замер на **живом аудио** (не синтетика): 20 секунд речи, записанных из
loopback (аудиокнига), прогнаны через три движка на одной машине.

## Методика
- Один и тот же файл `bench20.wav` (20.0 с, реальная речь) для всех движков.
- Прогрев (1 прогон) + 2 замеренных прогона, берётся **медиана**.
- Команда: `BrainstormBuddy.Headless.exe --bench <wav> <onnx> <labels> <dockerUrl>`.
- RTF = время обработки / длительность аудио (меньше = быстрее; <1 значит быстрее реального времени).
- Проверка корректности: все три движка выдали **одинаковое число символов (277)** →
  распознавание идентично (нативный порт численно совпадает с эталоном).

## Тестовая машина
Две видеокарты (важно для выводов):
- **GPU 0: Intel UHD Graphics** (встроенная, слабая) — DirectML device 0.
- **GPU 1: NVIDIA GeForce RTX 4050 Laptop** (дискретная) — DirectML device 1.

## Результаты (аудио 20.0 с)

| Движок | Медиана, мс | RTF | Символов |
|---|---:|---:|---:|
| 🥇 **Native GPU:1 — RTX 4050** (DirectML) | **381** | **0.019** | 277 |
| Native CPU (GigaAM ONNX) | 1084 | 0.054 | 277 |
| Docker (remote GigaAM) | 1211 | 0.061 | 277 |
| Native GPU:0 — Intel UHD (DirectML) | 1537 | 0.077 | 277 |

## Выводы
1. **Дискретный GPU (RTX 4050) — вне конкуренции: 381 мс на 20 с речи = RTF 0.019,
   ~52× быстрее реального времени, в ~3× быстрее CPU и Docker.**
2. **Встроенный GPU (Intel UHD) МЕДЛЕННЕЕ CPU** (1537 vs 1084 мс). Отсюда правило:
   **дискретка — да, встройка — нет**. GPU нельзя включать вслепую по индексу 0.
3. **Выбор видеокарты обязателен** (у юзеров бывает 2 GPU). В настройках — пикер GPU
   + режимы CPU/GPU/Авто; «Авто» выбирает дискретную карту, иначе CPU.
4. **Native CPU** (1084 мс) — надёжный универсальный дефолт, чуть быстрее Docker и без
   зависимости от Docker/Python (in-process, ~1 ГБ меньше диска/RAM).
5. Все варианты сильно быстрее реального времени — для живого суфлёра задержки STT нет.

## Важные оговорки
- Индекс DirectML-устройства = порядок DXGI-адаптеров (здесь 0=встройка, 1=дискретка;
  подтверждено замером: device 1 дал 381 мс). На другой машине порядок может отличаться —
  поэтому в приложении есть **пикер GPU по имени** и **кнопка замера скорости**, чтобы
  юзер проверил и выбрал сам.
- Модель: GigaAM v2 CTC, ONNX fp32 (932 МБ). fp16 (в 2× меньше) для GPU можно
  доэкспортировать с CUDA-машины — на CPU это не даст ускорения.
- Docker-путь оставлен как fallback (внешний сервер на отдельной машине — валидный сценарий).

_Замер выполнен командой `--bench` из Headless; сырые тайминги — в логах приложения._

# E2E на реальных интервью: GigaAM vs Whisper, живой режим (2026-07-12)

Сквозной прогон на трёх реальных видео-интервью (локально, офлайн, звук с машины
никуда не уходит) через `--endpoint-e2e <mp4> --engine both`. Это отвечает на вопрос
не «быстро ли ядро», а «какой движок годится для **живого** режима, где важна задержка
на каждую реплику».

## Что такое «чанк»
Приложение не гонит звук в STT непрерывно. Оно слушает и, когда собеседник делает
**паузу** (адаптивный эндпойнтинг «авто по паузам», `AudioBuffer` + `PauseAdaptiveController`),
отрезает кусок речи — **чанк = одна реплика/фраза**. Каждый чанк отдельно распознаётся →
текст на оверлее. Значит важна задержка «конец фразы → текст», т.е. **время на 1 чанк**.

## Результаты (окно 300с, эталон — Whisper medium, не истина)

| Видео | движок | чанков | точность | STT-время | RTF | ≈ на 1 чанк |
|---|---|---:|---:|---:|---:|---:|
| стартап | GigaAM | 15 | 76.5% | 6с | **0.02** | **~0.4с** |
| стартап | Whisper GPU | 15 | 79.2% | 200с | 0.67 | ~13с |
| интервью-2 | GigaAM | 28 | 77.0% | 16с | **0.05** | **~0.6с** |
| интервью-3 | GigaAM | 40 | 73.3% | 23с | **0.08** | **~0.6с** |
| интервью-3 (60с) | Whisper GPU | 12 | 51.7% | 154с | 2.57 | ~13с |

## Ключевой вывод: почему Whisper «лагает», а GigaAM — нет
Разложив время по двум прогонам (`15C+300p=200`, `12C+60p=154`): **фиксированная цена
≈ 12.5с на чанк** + всего **0.033–0.04 RTF на секунду аудио**. То есть само распознавание
на GPU — быстрое; дорога́ **фиксированная надбавка на каждый чанк**.

Причина — **архитектура самой сети Whisper**: `whisper_full` всегда кодирует окно
**30 секунд** на каждый вызов, даже если реплика 3 секунды (добивает нулями до 30с).
На коротких живых чанках это ~12с/реплику даже на дискретном GPU (на CPU — 30с+, отсюда
жалобы на «полминуты»). Это НЕ баг и НЕ наш эндпойнтинг (он режет паузы корректно —
в отчётах видно `adaptive→0.9с`), а свойство модели. Проверено: переиспользование
`WhisperProcessor` между чанками цену НЕ снижает (те же ~12.5с) — правка откачена.

**GigaAM** (CTC) кодирует ровно поданную длину → ~0.5с на реплику, RTF 0.02–0.08.

## Практический итог
- **Живой режим → GigaAM** (дефолт `SttEngine = "native"`): ~0.5с на реплику, «как t-one».
- **Whisper → транскрибация файлов** (длинное непрерывное аудио, мало границ окна) и
  английский/пунктуация. Для живого он медленный по своей природе.
- Переключение движков gigaam↔whisper на лету работает (грузится → выгружается →
  грузится) без краша — проверено сквозным прогоном.

_Замер: `BrainstormBuddy.Headless.exe --endpoint-e2e videos/<mp4> --engine both --maxsec N --whisper-accel gpu`; отчёты в `%APPDATA%\BrainstormBuddy\eval\`._

## Авто-калибровка порога на тихом источнике (2.5.4)

Симптом «STT перестаёт распознавать через ~5 минут» на тихом loopback/видео. Прогон
реального аудио (интервью, 15 мин) через `AudioBuffer`, отмасштабированного до разных
уровней громкости, с подсчётом эмитнутых чанков по минутам (`MinSpeech=400` в обоих режимах,
чтобы изолировать вклад именно порога):

| Уровень (глоб. RMS) | Режим порога | Чанков за 15 мин | «Мёртвые» минуты |
|---|---|---|---|
| **0.004** (тихий loopback) | **Авто-калибровка** | **73** | нет |
| 0.004 | Фиксированный (0.01) | 31 | мин. 1, 9, 12, 15 |
| 0.006 | Авто | 33 | нет |
| 0.006 | Фиксированный | 40 | мин. 1 |
| 0.008 | Авто | 27 | нет |
| 0.008 | Фиксированный | 59 | мин. 1 |

**Вывод.** На тихом уровне (тот, что реально был при захвате видео через динамики) фикс-порог
проваливается в 0 на нескольких минутах — это и есть «тыква». Авто-калибровка держит поток
ровно все 15 минут (ни одной мёртвой минуты) и даёт вдвое больше чанков (73 vs 31). На громких
уровнях оба режима работают без провалов — авто ничего не ломает. Дефолт — авто ВКЛ; ручной
порог остаётся в Настройках для тонкой подстройки. Юнит-тесты: `AudioBufferTests.AutoCalibration_*`.

## Повторный замер после фиксов 2.5.6 (защёлка + латч-брейкер): авто vs ручные пороги

То же 15-мин реальное аудио на трёх громкостях. «Мёртвая минута» = минута без единого чанка.
forced = склейки по 60с (фраза не закрылась паузой), latch = принудительная нарезка сплошного звука.

| Уровень | АВТО (2.5.6) | Ручной 0.010 (старый дефолт) | Ручной 0.005 |
|---|---|---|---|
| 0.004 (тихий loopback) | **93 чанка, 0 мёртвых** | 31, **4 мёртвых минуты** | 59, 1 мёртвая |
| 0.008 (средний) | **68, 0 мёртвых** | 59, 1 мёртвая | 87, 0 |
| 0.020 (громкий) | **56, 0 мёртвых** | 77, 0 | 29, из них 4 forced-склейки |

Дополнительно: латч на нормальной речи **не сработал ни разу** во всех 9 прогонах (фича «авто
по паузам» не затронута — фразы закрываются паузами); медианная задержка закрытия фразы
одинакова во всех режимах (= порог тишины, 1.8с в тесте); короткая реплика 0.6с эмитится через
те же 1.8с после конца речи и в авто, и в ручном.

**Вывод.** Авто-калибровка (после фиксов) — единственный режим без мёртвых минут на ВСЕХ
громкостях: ни один фиксированный порог не покрывает диапазон тихое-видео ↔ громкий-звук
одновременно (0.010 глохнет на тихом, 0.005 даёт 60с-склейки на громком). Цена по задержке —
нулевая. Проблемы 2.5.5 были багами реализации (пол-минимум, заморозка пола), оба закрыты
регресс-тестами (`AutoCalibration_SpikyNoise_*`, `ContinuousSound_LatchBreaker_*`).
