# План: авто-эндпойнтинг речи (адаптивный + семантический) и его тест-харнесс

Ветка `feature/agent-eval`. Задача: заменить ручной выбор темпа речи (быстрая/стандарт/
медленная) авто-режимами, которые сами подстраивают порог нарезки реплик под говорящего.
Юзер выбирает режим в настройках. Всё — офлайн, на локальном железе, без новых нативных
зависимостей.

Документ основан на: (а) эмпирике пауз из 3 реальных видео, (б) многоагентном дизайн-проходе
с адверсариальной критикой каждого подхода. Приватные видео из `videos/` в git не попадают
(`.gitignore`), в отчётах — только агрегаты.

---

## 1. Что такое «эндпойнтинг» и почему это узкое место

Живой поток: аудио → VAD режет на чанки по **паузе тишины** (`SilenceSeconds`) → STT → агенты.
`SilenceSeconds` — длина паузы, после которой реплика считается законченной. Ровно этот
параметр крутят три ручные кнопки в оверлее (`OnSilenceSpeedDown/Up/ResetSpeed`,
[OverlayWindow.xaml.cs:621](BrainstormBuddy/OverlayWindow.xaml.cs:621)) и пресеты в настройках
([SettingsViewModel.cs:264](BrainstormBuddy.Core/Config/SettingsViewModel.cs:264)).

- Порог **слишком длинный** → фразы слипаются в один огромный чанк, ответ приходит поздно.
- Порог **слишком короткий** → реплику режет на середине мысли, агент отвечает на обрывок.

Угадать нельзя — темп у людей разный. Это классическая задача *adaptive endpointing / turn
detection*.

### 1.1. Эмпирика (замер на 3 видео, ffmpeg silencedetect −40 dBFS = боевой RmsThreshold 0.01)

Распределение пауз (медиана / p85 / p95):
- ИНТЕРВЬЮ-2: 0.40 / 0.82 / 1.27 с (1086 пауз/52 мин)
- ИНТЕРВЬЮ-3: 0.41 / 0.87 / 1.43 с (1056 пауз/48 мин)
- СТАРТАП: 0.47 / 0.89 / 1.76 с (616 пауз/27 мин)

Симуляция эндпойнтинга (`ChunkMaxSeconds=15` — принудительная «гильотина»):

| Режим | % чанков — гильотина 15 с | медиана чанка |
|---|---|---|
| фикс 4.0 с (дефолт кода) | **90–98%** | 15.0 с |
| фикс 1.8 с (боевой config) | **71–91%** | 15.0 с |
| фикс 0.8 с | 28–38% | 5–9 с |
| фикс 0.6 с | 12–20% | 3–6 с |
| АВТО (корректная выборка) | 34–40% | 9–10 с, T сходится к ~0.9 с |
| АВТО (цензур. выборка — БАГ) | 5–13% | 3–4 с, T рушится в пол 0.5 с |

**Два вывода, оба критичные:**
1. **Текущие дефолты катастрофичны.** При боевых 1.8 с (и тем более дефолтных кода 4.0 с)
   реальные паузы почти не ловятся: 71–98% реплик режутся тупой 15-секундной гильотиной.
   Это и есть корень жалоб на «слипание». Эмпирический оптимум для этих спикеров — **0.6–0.9 с**.
   → Отдельная правка (независимо от авто): **понизить дефолт `SilenceSeconds` до ~1.0 с**.
2. **Наивная адаптация расходится** (подтверждено численно и обеими критиками): если в выборку
   темпа брать только паузы короче текущего порога, перцентиль занижается → порог падает в пол →
   жёсткая фрагментация, из которой система не выходит. Это **поглощающее состояние** —
   центральный риск, обязателен анти-коллапс (см. §2.4).

## 2. Подход 1 — адаптивный по распределению пауз (основной, реализуем полностью)

Дёшево, офлайн, без LLM. Оценивает темп говорящего по его же паузам и ведёт порог сам.

### 2.1. Что копим — «внутриречевые паузы»
Внутриречевая пауза (gap) — серия silence-кадров при `_isSpeaking==true`, **после которой
речь возобновилась** до срабатывания порога. Это выборка реальной паузной привычки оратора.
Точка захвата — ветка возобновления речи в `ProcessFrame`
([AudioBuffer.cs:253](BrainstormBuddy.Core/Audio/AudioBuffer.cs:253)), прямо перед вливанием
`_silenceTail`, где доступен `_silenceFrameCounter` (= длина паузы в 30-мс кадрах).

### 2.2. Оценщик
Кольцо последних `Window=48` gap'ов (мс). Порог-цель:
```
rawTarget = max(p85 × Multiplier, p85 + MarginMs)   // Multiplier=1.5, MarginMs=350
```
p85 считается nearest-rank по `stackalloc`-копии (48 int — наносекунды, вызывается редко).
`Multiplier > 1.2` — не косметика, а **стабилизатор**: гарантирует, что неподвижная точка
петли лежит выше сэмплируемой массы (восстанавливающая обратная связь).

### 2.3. Устойчивость (4 демпфера против осцилляции)
EMA цели (α=0.25) → deadband 150 мс → dwell 3000 мс между применениями → step-cap 0.3 с/шаг →
клампы коридора **[0.6, 2.2] с** (ручной режим сохраняет широкий [0.4, 10]).

### 2.4. ⚠ Обязательные правки из критики (без них авто ВРЕДИТ)
Дизайн «годен с правками». Эти пункты — блокирующие, заложены в реализацию:

1. **Анти-коллапс (S1, подтверждён эмпирикой).** Выборка resume-пауз систематически лишена
   длинных пауз (они становятся эндпойнтами и исчезают). Защита: (а) паузы брать из
   **независимой сырой VAD-маски** `SimpleVad.IsSpeech` (не из эмитов буфера — тогда наблюдения
   не цензурируются); (б) финальную паузу учитывать как **право-цензурированную** `≥T`; (в) если
   верх выборки «тонкий» и T стоит на полу — **дрейф T вверх к cold-start** (пробинг), чтобы не
   застрять на полу. Двухголосое интервью (пауза одного забита речью другого) → тот же коллапс,
   тот же guard + клампы.
2. **Чистота выборки (S2).** Ветка resume НЕ требует `MinSpeechFrames` → шумовой блип (музыка,
   щелчок) пишет фиктивный gap и сбрасывает счётчик тишины. Требовать **N подряд speech-кадров**
   (как на старте реплики) перед `RecordGap`; блипы 1–2 кадра игнорировать.
3. **Потокобезопасность контроллера (C1).** Его трогают 3 потока (mixer под `_lock`, STT-worker
   БЕЗ лока, UI-таймер). Внутренний lock/`volatile`/`Interlocked`; порядок захвата — никогда не
   брать `AudioBuffer._lock` под локом контроллера.
4. **`UpdateConfig` → `_micBuffer` (F2).** Сейчас
   [AudioCaptureEngine.cs:348](BrainstormBuddy/Audio/AudioCaptureEngine.cs:348) обновляет только
   `_buffer` — mic-канал заморожен (существующий баг). Пробросить и режим, и параметры в оба.
5. **Честный прогрев (W1).** До 8 gap'ов — не «10–20 с», а **1.5–4 мин** (пауз-resume 2–5/мин).
   Значит критичен **тёплый старт**: персистить выученное значение на канал.
6. **Выученное состояние — в отдельный state-файл (W3),** не в редактируемый юзером `config.json`.
7. **Ручной override без противоречий (W2).** Кнопки темпа в авто-режиме = ±bias к коридору
   (аддитивный, персистентный), STT-темп — отдельное слагаемое. Не оставлять «мёртвых» слайдеров.

### 2.5. По-канально
Loopback (интервьюер) и микрофон (кандидат) — **два раздельных `AudioBuffer`**
([App.xaml.cs:120](BrainstormBuddy/App.xaml.cs:120)) → у каждого свой контроллер с независимым
темпом. Даром получаем раздельную адаптацию под двух говорящих. MicOnly / single-buffer —
работает по имеющемуся каналу, в смешанном режиме точность ниже (документируем).

### 2.6. STT-темп (вторичный сигнал, опц.)
После чанка известны `clean.Text.Length` + `audioSeconds` → симв/с. Даёт (а) ускорение
холодного старта и (б) медленный bias коридора (±0.25 с). Перцентиль пауз самодостаточен как
основной драйвер; STT-темп — мягкая поправка, выключаемая.

### 2.7. Интеграция (точные правки)
Новый класс `BrainstormBuddy.Core/Audio/PauseAdaptiveController.cs` (чистый, юнит-тестируемый):
`RecordGap(int)`, `NoteTranscript(int,double)`, `TryConsumeChange(out double)`,
`SeedWarmStart(double)`, свойства `AppliedSeconds/SampleCount/LastP85Ms/IsWarm`.
`enum EndpointMode { Manual, AdaptivePauses, Semantic }`.
`AudioBuffer`: опциональные поля `_adaptive`/`_endpointMode`, методы
`EnableAdaptiveEndpointing(ctrl)`/`SetEndpointMode(m)`, захват gap'а в `ProcessFrame`,
`ApplyAdaptiveThresholdLocked(sec)` пишет `_silenceMs/_silenceFrameThreshold`. `HasCompleteUtterance`/
`TryGetReadyChunk`/`ExtractWavLocked`/`ResetStateLocked` — **не трогаем**. Adaptive off по
умолчанию → байт-в-байт старое поведение, `AudioBufferTests`/`SimpleVadTests` зелёные.
`AppConfig.AudioConfig` — новые поля с дефолтами (обратная совместимость `config.json`).

## 3. Подход 3 — семантический: ПЕРЕОСМЫСЛЕН (критика: «переделать»)

Исходная идея — LLM решает «мысль закончена?» перед нарезкой аудио. Критика показала, что
**чистый pre-emit семантический эндпойнтинг в дефолтной конфигурации нежизнеспособен**, и это
важный инженерный вывод (фиксируем, чтобы не наступить снова):

- **GigaAM не выдаёт пунктуации** (`labels.json` = пробел + 33 строчные буквы). Любая эвристика
  на `?`/`.`/`,` мертва; вопросительные слова в русском стоят в НАЧАЛЕ клаузы, а не в конце.
- **LLM-классификатор через `AskAsync` мёртв**: rate-limit 15 с
  ([OpenAiClient.cs:167](BrainstormBuddy.Core/Ai/OpenAiClient.cs:167)), temperature захардкожена
  0.3, нет `stop`/`top_p`, и он **крадёт слот у ответа** (двигает `_lastAskUtc`).
- **Спекулятивный STT в критическом пути** добавляет свою длительность → ранний эмит ≈ жёсткий
  порог, выигрыш ≈ 0. На слабом CPU три STT/LLM дерутся за одни ядра.

**Спасаемая версия — текстовая «отложенная склейка» (turn aggregation) ПОСЛЕ STT:**
режем аудио по адаптивному порогу (подход 1), а *семантику выносим на уровень текста*:
накопленный STT-текст перед вызовом агентов пропускаем через **гейт завершённости**, который на
lowercase-тексте без пунктуации решает — слать сейчас или **придержать и склеить со следующим
фрагментом**:
- **эвристика (дефолт, без LLM):** обрыв на предлоге/союзе/частице (`и`, `но`, `что`, `чтобы`,
  `потому`, `который`, `а`, `или`…) в конце → мысль не закончена, ждём; вопрос — по лид-словам
  в начале клаузы (`что/как/сколько/почему…`).
- **LLM-ассист (опц., только когда LLM простаивает):** отдельный classify-метод в обход
  rate-limit, temp=0, свой короткий таймаут, НЕ трогающий `_lastAskUtc`; при недоступности/
  таймауте — деградация в эвристику. На whisper-движке (даёт пунктуацию) работает точнее.
- предохранители: макс. время удержания, кап на склейки, взаимодействие с `ChunkMaxSeconds`.

Это развязывает рискованное real-time аудио-решение и семантику: аудио режет подход 1, текст
агрегирует подход 3. Честно и практично. В UI подаётся как «Авто (умная склейка)».

## 4. Выбор режима юзером

Настройки → Звук → «Режим нарезки речи»:
- **Ручной** (как сейчас): пресеты быстрая/стандарт/медленная + слайдер.
- **Авто — по паузам** (подход 1): сам подстраивается под темп. Рекомендованный дефолт.
- **Авто — умная склейка** (подход 3): подход 1 + текстовый гейт завершённости.

`EndpointMode` в конфиге (`"manual"|"adaptive"|"semantic"`), дефолт на время обкатки — `manual`
(чтобы ничего не сломать у текущих юзеров), после валидации харнессом — переключим дефолт.
В оверлее показываем текущее авто-значение порога (не «чёрный ящик»).

## 5. Тест-харнесс эндпойнтинга (доказываем цифрами, без человека и без живого звука)

Идея: `MediaAudioExtractor.Extract(mp4)` → float[] 16 кГц → кормим **настоящую** VAD-стейт-машину
`AudioBuffer` кусками → собираем эмитнутые чанки с точными границами. Детерминированно,
идентично проду (стейт-машина пофреймовая). Обязателен фейковый `TimeProvider` (уже поддержан
в ctor, [AudioBuffer.cs:140](BrainstormBuddy.Core/Audio/AudioBuffer.cs:140)).

### 5.1. Инструментализация (аддитивно, ~15 строк в Core)
Перегрузка `TryGetReadyChunk(out byte[] wav, out ChunkInfo info)` с
`ChunkInfo(OnsetSample, EndSample, Reason)` где `Reason ∈ {silence, forced, flush}`; поля
`_consumedSamples`/`_onsetSample` инкрементятся в `AddSamples`. **Починить `_forcedEmits`**:
сейчас инкрементится только в ветке «не говорим»
([AudioBuffer.cs:298](BrainstormBuddy.Core/Audio/AudioBuffer.cs:298)), а форс-во-время-речи
(305–320) считает только `_utterancesEmitted` — 2 строки правки. Старый 4-арг ctor и сигнатуры —
без изменений.

### 5.2. Эталон и метрики
Эталон — `videos/*_timestamped.txt`, **реальный формат `mm:ss  текст`** (проверено чтением; НЕ
`[mm:ss]`), парсер поддерживает и `[mm:ss]`/`[h:mm:ss]` для совместимости. Границы реплик = смена
временной метки.

Метрики (с guard'ами против NaN — вырожденные `nevercut`/тишина не должны ронять калькулятор):
- **Фрагментация F** — граница чанка попала внутрь эталонной реплики (рез мысли).
- **Слипание M** — один чанк накрыл > K эталонных реплик.
- **Латентность эндпойнта** — от фактического конца речи (из `ChunkInfo`) до emit.
- **% forced** (по `ChunkInfo.Reason=="forced"`, не по битому счётчику) и % suppressed.
- **Доля микро-чанков** (`dur<1 с`) — против «смерти от тысячи микро-резов».

Единая метрика Q (взвешенная) — но с защитой от гейминга (критика 5.1–5.6): `fixed:0.4` и
`nevercut` обязаны лежать на дне; штраф за суб-порог при высокой F; `Σ`(suppressed) — в
диагностику, не в Q (конфаунд с `MinSpeechMs`). Эталонная VAD-маска и тестируемый эндпойнтер
не должны делить один `vadMode` — прогнать чувствительность.

### 5.3. Честность сравнения
Эталон сам машинный и дробный — не «истина», а прокси. Поэтому: не подгонять Q под него;
семантический режим в общем рейтинге Q мерить осторожно (его реальная латентность = детект +
STT + judge, в аудио-времени не видна — выносим отдельно). Вывод «авто лучше лучшей ручной
настройки на X%» — только если он устойчив к выбору `vadMode` и весов.

### 5.4. Команда
`Headless --endpoint-bench <folder|mp4> [--modes fixed:0.6,fixed:1.8,adaptive,semantic] [--vad N]`
→ отчёт `csv`+`md` в `%APPDATA%\BrainstormBuddy\eval\` (вне репозитория). В отчёт не попадает
текст видео — только границы, длительности, счётчики.

## 6. Порядок работ

1. **Инструментализация Core** (`ChunkInfo`, фикс `_forcedEmits`) + `--endpoint-bench` симулятор
   + метрики → прогон фикс-режимов на 3 видео. *Первый цифровой результат: подтвердить, что
   текущие 1.8/4.0 с плохи, а 0.6–0.9 с хороши.*
2. **`PauseAdaptiveController`** (+ юнит-тесты на сходимость и анти-коллапс) → добавить `adaptive`
   в бенч → доказать, что авто ≈ лучшей ручной настройке без подбора юзером.
3. **Интеграция adaptive в `AudioBuffer`/`App`/`UpdateConfig`** (opt-in, тесты зелёные) +
   тёплый старт + per-channel.
4. **Текстовый гейт завершённости** (подход 3, эвристика) → режим `semantic` в бенче.
5. **UI:** режим в настройках + показ авто-порога в оверлее. Понизить дефолт `SilenceSeconds`.
6. Прогон всех режимов на 3 видео, свод в `summary.md`, вывод по дефолту.

## 7. Критерий приёмки
Режим/правку принимаем, только если на 3 видео: ↓ % forced-гильотины, ↓ фрагментация, латентность
в коридоре, и вывод устойчив к `vadMode`/весам. Иначе — откат. Дефолт не меняем, пока авто не
побьёт лучший ручной пресет численно.

## 8. Результаты (реализовано и измерено)

Прогон на настоящем VAD (`Headless --endpoint-bench videos`), полные видео:

| видео | fixed:1.8 (боевой) | fixed:0.8 (лучший ручной) | **adaptive (авто)** |
|---|---|---|---|
| ИНТЕРВЬЮ-2 | forced **90%**, покрытие 9% | forced 23%, micro 11% | →1.1с: forced **29%**, micro **7%**, покр. 17% |
| ИНТЕРВЬЮ-3 | forced **74%** | forced 21%, micro 10% | →0.9с: forced **18%**, micro 12%, покр. 18% |
| СТАРТАП | forced **87%** | forced 27%, micro 14% | →1.0с: forced **26%**, micro 13%, покр. 10% |

**Вывод:** адаптив сам подбирает порог под спикера (1.1 / 0.9 / 1.0 с) и **матчит или бьёт лучший
ручной 0.8 с при меньшей микро-фрагментации**, радикально уступая только по «полноте реплик»
катастрофичному боевому 1.8 с (который режет 74–90% тупой 15-секундной гильотиной). Юзеру не
надо угадывать. Латентность эндпойнта ~0.8 с (против произвольных резов гильотины).

Текстовая склейка (подход 3) на реальных фрагментах эталона склеивает **7–12%** оборванных на
союзе/предлоге мыслей — меньше пустых вызовов агентов на обрывках.
