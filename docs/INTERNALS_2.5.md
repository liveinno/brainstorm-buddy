# Внутренности 2.5.3–2.5.8: механизмы, решения, уроки

Сводка того, что менялось в аудио/LLM-конвейере за релизы 2.5.3–2.5.8 и ПОЧЕМУ.
Детали по темам: [AUDIO_CHUNKING.md](AUDIO_CHUNKING.md) (нарезка звука),
[LOGS_GUIDE.md](LOGS_GUIDE.md) (чтение логов), [STT_BENCHMARK.md](STT_BENCHMARK.md) (замеры),

## Конвейер «река» (с 2.5.3)

```
Аудио (WASAPI loopback + микрофон, раздельно)
  → AudioBuffer (VAD + нарезка: пауза / латч 10с / потолок 60с)
  → MainLoop: чанк → очередь STT (bounded 64)          [фиолетовая отсечка на шкале]
  → STT-воркер: GigaAM → текст → ПУЗЫРЬ СРАЗУ (с «ожидание»)
  → очередь ответов (bounded 32)
  → Answer-воркер: LLM → ответ подставляется в тот же пузырь
```
Ключевой принцип: STT никогда не ждёт LLM. Медленный/лежащий LLM не останавливает транскрипцию.

## Пороги речи: история и текущее состояние

| Версия | Состояние | Чем кончилось |
|---|---|---|
| ≤2.5.2 | Фикс-порог 0.01 (+ MinSpeech 2000мс) | «Тыква»: тихое видео (~0.004) не ловилось, короткие реплики выбрасывались |
| 2.5.3 | MinSpeech 400, Silence 1.8 | Тихое видео всё ещё ниже 0.01 |
| 2.5.4 | Авто-калибровка v1 (пол-минимум) | Фантомная «речь» на рваном шуме |
| 2.5.5 | Пол по EMA + заморозка при речи | Защёлка: непрерывный звук намертво открывал фразу |
| 2.5.6 | Дрейф пола вверх при речи + латч-брейкер | Работает, но авто-порог на живом железе удивлял (0.018 против речи 0.009) |
| **2.5.7+** | **Дефолт: авто ВЫКЛ, ручной порог 0.001** | Решение владельца по замерам: шум реального микрофона ≈ 0.000, речь ≈ 0.009 — ручной 0.001 предсказуем и ловит всё. Авто-калибровка осталась опцией для шумных сред |

Миграции конфига: v4 (MinSpeech/Silence/мульти-агент выкл), v5 (авто-порог выкл у всех,
0.01→0.001 для старого дефолта; кастом не трогается).

## LLM: честность проверки и живучесть ответов (2.5.7–2.5.8)

- «Проверить подключение» гоняет запрос с БОЕВЫМИ параметрами (тот же max_tokens и системный
  промпт) и требует непустой content. HTTP 200 ≠ успех.

### Reasoning-модели: почему «ответ пустой», хотя модель живая

«Думающие» модели (hy3/hunyuan/gpt-oss/o1/o3/deepseek-r/qwq/…) прежде чем ответить, пишут
**невидимые размышления** — черновик, который в content не попадает, но **съедает тот же лимит
токенов**, что и видимый ответ. Если лимит маленький (например, 180 токенов), модель успевает
только «подумать»: HTTP 200, usage насчитан, а видимый ответ — пустая строка. Выглядит как
баг приложения, на деле — устройство модели.

Наши фиксы: таким моделям автоматически ставится `reasoning: effort=low` («думай короче»);
если 200 всё равно пришёл с пустым content — авто-ретрай с лимитом до 1024 токенов. Ошибка
юзеру показывается только если и это не помогло.

**Не путать с лимитами тарифа.** Ошибки 402 («Prompt tokens limit exceeded», кончились кредиты)
и 429 (rate-limit, «try again in N seconds») — это ограничения БЕСПЛАТНОГО тарифа OpenRouter,
а не баг ПО: провайдер отказал до того, как модель начала отвечать. С 2.5.9 в плашке первой
строкой идёт человеческое объяснение («Лимит бесплатного тарифа OpenRouter исчерпан — пополните
счёт или переключитесь на локальную модель» / «Провайдер ограничил частоту запросов — подождите
или смените модель/тариф»), сырой текст провайдера — второй строкой (`OpenAiClient.HumanizeApiError`).
Прочие коды (401/404/…) показываются как есть.

Каждый запрос/ответ/ошибка LLM виден строкой в Настройки → Диагностика → «LLM-лог текущей
сессии» (событие `OpenAiClient.Exchange`, ~200 последних записей).

## Аудио-устройства (2.5.7)

- Loopback всегда пишет с ТЕКУЩЕГО системного устройства вывода. `IMMNotificationClient`
  следит за сменой дефолта (Bluetooth-наушники!) → авто-рестарт захвата (дебаунс 1.5с).
- Микрофон — только явный выбор юзера, автопереключения нет.
- Золотистый тост (CoachGold #F4C542, 25с, крестик) сообщает о смене вывода/новом устройстве.
- ~~Известный долг: выпадашка «Динамик (loopback)» декоративная~~ — закрыто в 2.5.9:
  `LoopbackDeviceId` честно читается (`AudioCaptureEngine.CreateLoopbackCapture`), выбор
  устройства работает, автоследование за системным дефолтом — только при «По умолчанию».

## Диагностика: как не потерять сессию (урок 2.5.5–2.5.6)

- **`C:\ProgramData\BrainstormBuddy\last-session.txt`** — крошка при каждом старте: PID,
  версия, exe, РЕАЛЬНЫЕ пути AppData/лога. Первая точка входа при любом «логов нет».
- Урок: процессы под MSIX/sandbox-виртуализацией видят СВОЮ копию %APPDATA% — файлы сессии
  могут лежать не там, где ищешь. Обход при чтении с виртуализированного процесса:
  UNC-путь `\\localhost\c$\...`.
- Логгер флашит построчно (буферизации НЕТ); при недоступности файла — 3 отказа →
  фолбэк в `%TEMP%\BrainstormBuddy\logs` + громкая пометка в ring-буфере.
- `Overlay emitted [канал] chunk #N` в логе = реплика дошла до экрана (путь STT→UI
  раньше был слепой зоной).

## Текстовый пост-фильтр (2.5.7–2.5.8)

Clean (повторы/паразиты) больше НИЧЕГО не гейтит: всё распознанное показывается И уходит
в LLM (решение владельца; реакцией на короткое управляет системный промпт).
Стоп-лист паразитов — см. AUDIO_CHUNKING.md; «да/нет/ага/угу» из него убраны.
