# Roadmap: нативный GigaAM (ONNX) + инсталлятор + LLM-провайдеры + UX

Ветка: **`feature/native-stt-onnx`** (от `main`).

## Правила работы
- **MR — только по явному указанию пользователя.** Не мержить в `main` без команды.
- Чистые клон-сборки не гоняем (долго). Обычный `dotnet build` для проверки компиляции — да.
- **Docker/`LocalSttBridge` НЕ удаляем** — остаётся рабочим движком. Всё «добавляем», не «заменяем».
- Только проверенные NuGet-пакеты (Microsoft.ML.OnnxRuntime и т.п.). Никаких сторонних репозиториев.
- Перед пересборкой закрывать запущенный `BrainstormBuddy.exe` (иначе MSB3027).

---

## Фаза 0 — Абстракция движков STT (низкий риск, без смены поведения)
- Интерфейс `ISttEngine { Task<string?> TranscribeAsync(byte[] wav, ct); Task<bool> HealthAsync(); string Name; }`.
- `RemoteSttService : ISttEngine` — обёртка над текущим `OpenAiClient.TranscribeAsync` (Docker/OpenAI-совместимый путь 1:1).
- Конфиг `Audio.SttEngine = "remote" | "native"` (дефолт `remote` — поведение не меняется).
- App выбирает движок; `WorkerLoop` зовёт `SttEngine.TranscribeAsync` вместо прямого `ApiClient.TranscribeAsync`.
- **DoD:** работает как сейчас, тесты зелёные.

## Фаза 1 — GigaAM нативно (ядро, ~1 неделя, CTC)
- **1.1 (Python, разово у нас):** `model.to_onnx(...)` → `gigaam_ctc.onnx` (fp16) + словарь + параметры фронтенда; снять эталон признаков/выхода на 2–3 wav.
- **1.2:** `NativeGigaamSttService : ISttEngine`:
  - мел-фронтенд как в NeMo (валидировать против эталона — главный риск);
  - `Microsoft.ML.OnnxRuntime` (+ DirectML EP: AMD/Intel/NVIDIA → CPU fallback);
  - CTC-декод (argmax → схлопнуть повторы → убрать blank → детокен по словарю);
  - чанкинг 24с/overlap 2с (из `LocalSttBridge/main.py`).
- **1.3:** авто-скачивание модели в `%APPDATA%\BrainstormBuddy\models\` с прогрессом.
- **1.4:** Настройки → Локальный STT: выбор движка, «Скачать модель», ускорение (Авто/DirectML/CPU) + детект GPU.
- **DoD:** транскрипт без Docker; переключение на Docker работает; регресс зелёный.
- Опционально: RNNT (`gigaam-e2e`) — качество выше, +2–4 дня.

## Фаза 2 — UX-плюшки
- **2.1 Замер скорости STT** в настройках: прогон эталона через активный движок → мс + RTF + вердикт (🟢<0.3 / 🟡 / 🔴>0.7). Переиспользует инфру теста микрофона.
- **2.2 Первый старт/онбординг:** флаг `firstRun`; проверка Ollama, наличия модели, детект GPU, предложить скачать модель.
- **2.3 Уведомления** (расширяем баннер `HealthChanged`): «модель не скачана», «GPU не найден — CPU (медленнее)», «Ollama не запущен», «нативный движок упал → переключился на внешний».

## Фаза 3 — LLM-провайдеры (низкий приоритет)
- Облако ломает УТП приватности → по минимуму, опционально.
- Пресеты OpenAI-совместимых (Ollama/OpenRouter/Groq/LM Studio): `BaseUrl` + ключ (код-путь уже есть) + кнопка «Проверить связь» (`CheckConnectionAsync`).
- Claude — не OpenAI-формат → отдельный адаптер только если реально нужен. 100% out-of-box не обещаем, ключи не покупаем.
- Дефолт — локальный Ollama.

## Фаза 4 — Инсталлятор (в конце)
- Сначала закрыть `PUBLISH_ISSUE_NOTES.md` (single-file publish).
- Velopack → `Setup.exe` + автообновления (или Inno Setup). Модель/GPU-рантаймы — докачка по требованию.
- Подпись кода (SmartScreen). CI: тег → сборка в релизы. **НЕ MSIX** (песочница ломает стелс).

### Ключевые факты порта (для будущих агентов)
- GigaAM v2 CTC: Conformer ~240М, **посимвольный** словарь (33 буквы + blank=33, vocab=34).
- Фронтенд = torchaudio MelSpectrogram: sr16000, n_fft400, win400, hop160, 64 мела, hann(periodic),
  power2, mel_scale htk, f_min0/f_max8000, norm None, затем ln(clamp(x,1e-9,1e9)). Совпал до 5e-5.
- ONNX вход: `features[1,64,T]` + `feature_lengths[1]`; выход `log_probs[1,T',34]` (T' ≈ T/4).
- Модель `v2_ctc.onnx` 932МБ fp32 (экспорт с CPU-контейнера; fp16 — только с CUDA-машины).
- Экспорт: `tools/gigaam_export/export_onnx.py` (запускать в контейнере с пакетом `gigaam`).
- Валидация: `Headless --validate-mel <dir>` и `--validate-onnx <dir> <onnx>`.
